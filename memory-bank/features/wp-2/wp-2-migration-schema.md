# РП 2: Базовая таблица `audit_events` и Ecto schema

Соответствует строке **РП 2** в [`WORKPLAN.md`](../WORKPLAN.md): первая миграция и минимальная схема для записи и чтения событий аудита.

## Цель

Заложить устойчивый контракт хранения: одна таблица событий и один модуль `Ecto.Schema`, с которыми можно выполнять `insert` и `get`/`all` через `Audit.Repo` без доменной логики библиотеки (она — в следующих РП).

## Объём

**Входит:**

- Первая миграция Ecto в приложении `audit` (каталог `priv/repo/migrations/`), таблица **`audit_events`**.
- Модуль schema (например `Audit.AuditEvent`) с `@primary_key` и полями, согласованными с миграцией; `use Ecto.Schema`; при необходимости — `Ecto.Changeset` только если нужен для валидации минимальных полей при insert в тестах.
- Конфигурация репозитория уже есть (`Audit.Repo`); при необходимости — правка `Audit.Application` для запуска Repo в `:dev`/`:test`, если миграции или тесты БД этого потребуют.

**Не входит (сознательно позже):**

- Публичный API библиотеки, автоматический перехват изменений, политики ретенции, сложные diff полей.
- Отдельные таблицы под «снимки» до/после, если решим хранить только JSON — достаточно одного расширяемого поля под полезную нагрузку.

## Таблица `audit_events` (предлагаемый минимум)

Имена и типы можно уточнить при реализации; ниже — ориентир для первой итерации.

| Колонка | Тип PostgreSQL | Назначение |
|--------|----------------|------------|
| `id` | `uuid` | Первичный ключ (генерация UUID на уровне Ecto/БД) |
| `operation` | `varchar`, `NOT NULL` | Вид операции: например `insert`, `update`, `delete` |
| `entity_type` | `varchar`, `NOT NULL` | Логическое имя сущности (таблица, тип ресурса и т.д.) |
| `entity_id` | `text`, `NOT NULL` | Идентификатор сущности в строковом виде (универсально для UUID, integer id, составных ключей как сериализованная строка) |
| `metadata` | `jsonb`, `NOT NULL`, default `'{}'` | Произвольный контекст (IP, correlation id, …) |
| `payload` | `jsonb`, `NULL` | Полезная нагрузка: снимок / дифф / произвольная структура под следующие РП |
| `inserted_at` | `utc_datetime_usec` | Системная колонка Ecto из `timestamps(updated_at: false)`; используется как момент события |

Для времени события используем системный подход Ecto: в миграции `timestamps(updated_at: false, type: :utc_datetime_usec)`, чтобы оставить только колонку вставки (`inserted_at`) и не хранить `updated_at`.

Индексы на первом шаге опциональны; разумный кандидат — `(entity_type, entity_id)` или `(inserted_at)` для выборок по сущности / времени.

## Критерии готовности

1. `mix ecto.migrate` в среде dev (при настроенной БД) применяет миграцию без ошибок.
2. Schema-модуль компилируется, поля совпадают с миграцией (включая `timestamps`/время события — единый подход с `config :audit, Audit.Repo, migration_timestamps: ...`).
3. Есть хотя бы один автоматический тест: вставка записи через `Repo.insert` и чтение по `id` (или простой `Repo.all` с фильтром), чтобы фиксировать контракт на уровне кода.

## Требования к миграции как интерфейсу библиотеки

Так как это библиотека, миграция должна быть доступна пользователю как стабильный интерфейс, а не только как внутренняя миграция demo-приложения.

Ожидаемый сценарий интеграции для пользователя библиотеки:

1. В своём приложении пользователь создаёт пустую миграцию (например, `add_audit_table`).
2. Внутри миграции делегирует создание/удаление таблицы в библиотеку:

```elixir
def up do
  Audit.Migrations.up()
end

def down do
  Audit.Migrations.down()
end
```

Требование к РП:

- Предусмотреть модуль `Audit.Migrations` с функциями `up/0` и `down/0`, которые можно безопасно вызывать из пользовательской миграции.
- Сигнатура и поведение должны быть задокументированы как публичный контракт библиотеки.
- Шаги подключения (установка зависимости, генерация миграции, вставка вызовов `Audit.Migrations`, запуск `mix ecto.migrate`) описать в `README.md` библиотеки.
