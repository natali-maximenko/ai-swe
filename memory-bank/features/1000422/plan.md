## 3. Архитектура решения (дизайн)

### 3.1. Поток данных

```
base_query
|> Audit.JsonLogicToEctoQuery.run(query_filters)   # список restrictions после decode
|> maybe_select(return_fields)
```

Семантика: внешний список — **OR**, внутренние списки — **AND**; выражения JSON Logic только внутри `value` у элементов с `"type": "QUERY"`. Цепочка соответствует spec § Реализация.

**`return_fields` и `select`:** при непустом `return_fields` предыдущий `select` в запросе **полностью заменяется**; в проекции остаются **только** поля из `return_fields` (в порядке списка или согласно соглашению в коде — зафиксировать в `@moduledoc`).

### 3.2. Модули

| Модуль | Ответственность | Спека |
|--------|-----------------|-------|
| **`Audit.Policeman`** | Фасад: нормализация пустых входов (`nil` / `[]`), вызов `Audit.JsonLogicToEctoQuery`, `maybe_select`, отсутствие исключений на допустимых типах | § Решение, § Требования 3–4, § Инварианты |
| **`Audit.JsonLogicToEctoQuery`** | Обход дерева JSON Logic в белом списке, построение `dynamic` / условий `where`, логирование путей к неподдерживаемым узлам, лимиты глубины/размера, семантика пустых групп | § Реализация, § Белый список, § Требования 2–3, § Инварианты 1–3 |
| **(внутренний)** утилиты подсчёта/обхода | Подсчёт узлов, глубины, формирование «пути» для логов | § Требования 3 |

Логирование: `Logger.warning/2` при неподдерживаемом узле, превышении лимитов, «пустой политике QUERY», пустом корневом OR без предикатов — по spec § Требования 3 и § Сценарии ошибок.

### 3.3. Ключевые инженерные правила

- Только **Ecto.Query** и **параметризованные** биндинги, без интерполяции недоверенных строк в SQL (spec § Требования 4).
- Лимиты: по умолчанию глубина **10**, размер дерева **100**; настраиваемые опции модуля (spec § Белый список JSON Logic).
- При превышении лимита: для **данного вызова** фильтр по `query_filters` **не применяется**; `return_fields` обрабатываются по матрице (spec § Требования 3, § Матрица).

## 4. Матрица входов → поведение (реализация `Audit.Policeman`)

Реализовать ровно четыре нетривиальных сочетания + базовый случай (spec § Реализация, § Матрица):

| query_filters | return_fields | Действие |
|---------------|-----------------|----------|
| пусто | пусто | Вернуть `base_query` |
| не пусто | пусто | Только фильтры |
| пусто | не пусто | Только `select` по полям |
| не пусто | не пусто | Фильтры + `select` |

Пустые группы OR/AND и «корневой OR без предикатов» — по нормам в spec § Матрица (ветвь с пустым AND → ложь; весь OR пустой → `where false` + warning).

## 5. Пошаговый план работ (фазы)

### Фаза A — каркас в `audit/`

- Подключить/проверить зависимости в [`mix.exs`](../../../audit/mix.exs): для конвертера нужны Ecto (уже есть); при необходимости для AC — тестовая БД как в [`config/test.exs`](../../../audit/config/test.exs).
- Создать каркас файлов `lib/audit/policeman.ex` и `lib/audit/json_logic_to_ecto_query.ex` (см. § 2.1); модули `Audit.Policeman` и `Audit.JsonLogicToEctoQuery` — рядом с остальными модулями аудита в `lib/audit/`.
- Публичный API: `apply_restrictions/3` с документированным контрактом в `@moduledoc` (включая норму про пустые группы OR/AND — одна формулировка, spec § Матрица; и правило **полной замены** `select` при непустом `return_fields`).
- **Трассировка:** spec § Контракт API, § Инварианты 4.

### Фаза B — `Audit.JsonLogicToEctoQuery` (ядро)

- Парсер структуры `restrictions`: только элементы `type: "QUERY"`, извлечение `value`.
- Рекурсивный обход JSON Logic с белым списком: `==`, `!=`, `>`, `>=`, `<`, `<=`, `!`, `and`, `or`, `not`, `var`, `in` (spec § Белый список).
- Сборка условий через `dynamic` и объединение в семантике OR(AND(...)).
- Обработка неподдерживаемых узлов и лимитов (spec § Требования 3).
- Юнит-тесты: **на каждый оператор** белого списка; сломанная нотация; проверка через `Ecto.Adapters.SQL.to_sql` и наличие фрагментов условий/биндингов (spec AC).

### Фаза C — `return_fields` и привязка полей

- `maybe_select`: строковые имена полей → атомы схемы, динамический `select`, который **полностью подменяет** прежний `select` в `base_query`; в результате в проекции только поля из `return_fields` (spec § Scope, § Реализация и принятое решение в разделе 8).

### Фаза D — интеграционные тесты

- Четыре интеграционных теста на сочетания параметров `Audit.Policeman.apply_restrictions/3` (spec AC).
- Интеграционный тест на фикстуру [`audit/test/fixtures/PDP_responce.txt`](../../../audit/test/fixtures/PDP_responce.txt): прочитать JSON, взять `restrictions` / `fields`, прогнать через `Audit.Policeman.apply_restrictions/3` (spec AC4).
- При необходимости — тест с реальной БД и схемой, фильтрация строк (spec AC: «интеграционные — через схемы и настоящий запрос»).

### Фаза E — завершение

- Проверка полного списка AC из spec § Acceptance Criteria.
- Минимальная вычитка `@moduledoc` и публичных `@doc` (spec § Scope).

### Верификация — окружение и прогон тестов

- Поднять окружение зависимостей (например БД для интеграционных тестов): **`docker compose up`** в каталоге, где расположен актуальный `compose`‑файл для этого проекта.
- В каталоге Mix‑проекта [`audit`](../../../audit) выполнить **`mix test`** и убедиться, что **все тесты проходят** без ошибок и падений.
- При расхождении с локальной средой зафиксировать в README или комментарии к compose, какие сервисы обязательны для полного прогона.
