# [Конвертер restrictions на Elixir]

## Цель
Конвертер должен обеспечить возможность применения restrictions из запросов к выборкам данных из БД 
во всех сервисах на Elixir.

## Решение

Написать библиотечный модуль `Audit.Policeman` (файл `lib/audit/policeman.ex`), с таким интерфейсом:
```elixir
@spec apply_restrictions(Ecto.Query, list | nil, list | nil) :: Ecto.Query 
Audit.Policeman.apply_restrictions(base_query, query_filters, return_fields)
```
`query_filters` — значение поля `restrictions` из ответа PDP после `Json.decode`: вложенный **список** (внешний уровень — группы OR, внутри — элементы с `"type": "QUERY"` и JSON Logic в поле `value`), как в [PDP_responce.txt](../../../audit/test/fixtures/PDP_responce.txt). Отдельно корневого «дерева JSON Logic» без обёртки PDP нет: выражения JSON Logic живут **внутри** `value` у QUERY.

`return_fields` — список названий полей из `fields` того же ответа (после decode).

Пример: [PDP_responce.txt](../../../audit/test/fixtures/PDP_responce.txt) — `restrictions` → `query_filters`, `fields` → `return_fields`.

apply_restrictions применяет ограничения к запросу и возвращаемым полям.

## Реализация

Структура `query_filters` — как у PDP: **список**, не корневая мапа JSON Logic. Базовый набор операторов внутри `value` (операторы сравнения, `and`, `or`, `in`, … — см. белый список). Разбор вложенного списка и преобразование выражений из полей `value` в условия Ecto — в модуле **`Audit.JsonLogicToEctoQuery`** (`lib/audit/json_logic_to_ecto_query.ex`).

Цепочка:

```
base_query
|> Audit.JsonLogicToEctoQuery.run(query_filters)
|> maybe_select(return_fields)
```

(`Audit.JsonLogicToEctoQuery.run/2` принимает `base_query` (Ecto.Query) и **список** `query_filters` в форме `restrictions` после decode.)

Если оба параметра не пустые, к запросу на Ecto применяются все фильтры и список полей сокращается до набора из return_fields.
Если передан только query_filters (не пустой), фильтры применяются к запросу, возвращаемые поля не трогаем.
Если передан только return_fields (не пустой), запрос не трогаем, меняем только возвращаемые поля.
Если оба параметра пустые, возвращаем исходный запрос.
Пусто для `query_filters`: `nil` или `[]` (другие формы не входят в контракт). Пусто для `return_fields`: `nil` или `[]`.

## Scope

### Входит

- Разбор структуры **`restrictions`**: внешний уровень — **OR**, внутренний — **AND** (как в ответе PDP).
- Преобразование выражений JSON Logic из ограничений типа **QUERY** в условия уровня Ecto-запроса (без генерации произвольной строки SQL как основного пути).
- **Привязка полей:** на вход конвертера передаётся список полей, уже преобразованных из camelCase, мы считаем все поля называются верно и присутствуют в схеме таблицы, над которой выполняется базовый запрос, единственное что должен обеспечивать модуль это привязку названий полей в формате понятном ecto (преобразовать строковые названия к атомам и встроить их динамически в вызов select/3)
- Тесты библиотеки: фикстуры JSON Logic и `restrictions` → проверка, что сформированный запрос соответствует ожидаемой семантике **OR(AND(...))** и операторам белого списка.
- Документация публичного API модуля (минимум `@moduledoc` и описание аргумента привязки полей).

### Не входит

- Изменение HTTP-контрактов сервисов на Elixir, OpenAPI.
- Внедрение конвертера в сервисы, это другая задача.


## Требования

### Функциональные

**1.** Библиотека принимает нормализованное представление ограничений QUERY (после извлечения из ответа PDP) и **список полей**, возвращает преобразованный (например через `dynamic` или список динамик для последующего `where`) `Ecto.Query`, результат, запрос остаётся валидным Ecto.Query.

**2.** Семантика комбинирования совпадает по `restrictions`: дизъюнкция групп, конъюнкция внутри группы.

**3.** Литералы и операции только из **белого списка**. **Неподдерживаемый узел** (оператор вне белого списка, неверная форма узла): запись в лог **уровня `:warning`**, в сообщении — **путь к узлу** (внутри выражения в поле `value`, дерево JSON Logic); **этот узел не транслируется в SQL**, родительские `and` / `or` обрабатывают остальные дочерние узлы. **Превышение лимитов глубины или размера дерева:** запись в лог **`:warning`**; **для данного вызова `apply_restrictions` фильтр по `query_filters` целиком не применяется** (к запросу добавляется только то, что следует из остальной логики вызова, например `return_fields`). **Контракт `apply_restrictions/3`:** при входах типов `Ecto.Query`, `nil | list` для `query_filters` (список в форме поля `restrictions` после `Json.decode`, см. выше), `nil | list` для `return_fields` **функция не выбрасывает исключений**; сбои разбора отражаются только логом и ослаблением условий, как выше.

**4.** Нет интерполяции недоверенных строк в SQL: только API Ecto.Query и параметры биндинга там.

**5.** Публичный API покрыт тестами на соответствие переложению JSON Logic → Ecto.

## Белый список JSON Logic (стартовый)

Минимум: `==`, `!=`, `>`, `>=`, `<`, `<=`, `!`, `and`, `or`, `not`, `var`, `in`.  
Расширение — отдельным изменением спеки после разбора реальных политик.

Лимиты: максимальная глубина (по умолчанию 10) и размер дерева (по умолчанию 100) — через опции модуля можно регулировать. При срабатывании лимита см. требование **3** (фильтр по `query_filters` для вызова не применяется).

## Матрица входов и исходов (Uniform)

Успех — валидные типы и согласованный список `restrictions` (и JSON Logic внутри `value` у QUERY); **ошибка** здесь не означает исключение из `apply_restrictions/3`, а описанное ниже ослабление или отсутствие фильтра.

| query_filters | return_fields | Исход |
|---------------|---------------|--------|
| пусто (`nil` или `[]` для `query_filters`) | пусто (`nil` или `[]` для `return_fields`) | **Исходный** `base_query` без изменений. |
| не пусто | пусто | К `base_query` применяются только фильтры; **select** не меняем. |
| пусто | не пусто | Запрос **не** сужаем по фильтрам; **select** задаём по `return_fields`. |
| не пусто | не пусто | Фильтры + **select** по `return_fields`. |

**Успех (семантика):** после вызова `Ecto.Query` остаётся валидным; при непустых фильтрах и отсутствии ослаблений по п.3 условие в SQL соответствует **OR(AND(...))** и белому списку.

**Пустые группы в структуре `restrictions` (PDP):**

- **Член OR с пустым внутренним AND** (нет ни одного предиката после разбора): этот член даёт **ложь** для своей ветви (`AND` пустой → не выполняется); на уровне OR такая ветвь **не расширяет** допустимые строки.
- **Корневой OR, в котором нет ни одного члена с хотя бы одним предикатом** (все ветви пустые): итоговое условие по фильтрам — **ни одна строка не проходит** (`where false` / эквивалент для данного запроса), плюс запись в лог **`:warning`** с причиной «пустая политика QUERY».

Поведение пустых групп дублируется в `@moduledoc` одной нормой, без альтернатив «пропуск или ошибка».

## Контракт API (черновик для реализации)

- **Вход:** `base_query` — исходный запрос (`Ecto.Query`); `query_filters` — **список** из поля `restrictions` ответа PDP после `Json.decode` (та же форма, что в [PDP_responce.txt](../../../audit/test/fixtures/PDP_responce.txt)); `return_fields` — список строк имён полей (как `fields` в PDP) или `nil`. Оба аргумента `query_filters` и `return_fields` могут быть `nil`.
- **Выход:** преобразованный `Ecto.Query`.

---

## 7. Инварианты

1. **Неподдерживаемый узел** → лог `:warning` с путём к узлу; узел не попадает в SQL; соседние ветки `and`/`or` сохраняются. **Превышение лимитов** → лог `:warning`; фильтр по `query_filters` **для этого вызова** не применяется целиком.
2. Без сырой конкатенации пользовательских/PDP строк в SQL: только Ecto.Query и биндинги.
3. Пустой внутренний AND внутри члена OR → ветвь ложна; корневой OR без ни одного предиката → результат фильтрации пустой, лог `:warning`. См. «Матрица входов и исходов».
4. `apply_restrictions/3` не выбрасывает исключений на допустимых типах входов (см. требование **3**).

## Сценарии ошибок и пограничные состояния

| Условие | Ожидаемое поведение |
|--------|----------------------|
| Оба параметра `query_filters` и `return_fields` пустые | Возврат **исходного** `base_query`. |
| `query_filters` с поломанной нотацией / неподдерживаемым узлом | Лог `:warning`, путь к узлу; **исключений нет**; неподдерживаемый узел не в SQL; см. требование **3**. |
| Превышена глубина или размер дерева | Лог `:warning`; **фильтр по `query_filters` для вызова не применяется**; исключений нет. |
| Корневой OR без ни одного предиката | Условие фильтрации **пустой результат**; лог `:warning`; исключений нет. |
| Пустой внутренний AND внутри члена OR | Ветвь ложна; остальные члены OR без изменений логики OR. |

**Успех:** валидный список `restrictions`, лимиты не превышены, белый список соблюдён → запрос с ожидаемой семантикой OR/AND; если передан непустой `return_fields`, применяется описанный выше `select`.
